# AIMA 用户验收测试操作手册
## K3S + VLLM + Qwen 系列模型验证

**版本**：UAT Round 1
**适用对象**：所有测试员
**测试轮次**：K3S 初始化 → VLLM 多模型部署 → 性能验证 → 局域网发现

---

## 一、测试员基本规范（必读）

在开始任何测试之前，请仔细阅读并遵守以下规则：

**你必须做的事：**
- 在你负责的机器目录下维护一个实时日志文件，格式见第七节
- 每完成一个步骤，无论成功还是失败，都必须记录到日志
- 真实操作、真实观察、真实记录——所有输出结果原样复制到日志
- 遇到障碍时，先记录完整的错误信息，再尝试分析原因
- 无法继续推进时，汇总当前状态并向团队领导上报，等待指示

**你绝对不能做的事：**
- 不能修改 AIMA 的源代码、配置代码或 catalog YAML 文件
- 不能跳过失败步骤假装成功（如果某步失败，必须如实记录）
- 不能进行任何模拟或 mock——所有操作必须真实执行
- 不能自行安装或修改与 AIMA 无关的系统级服务（不要自行安装 K3S、HAMi 等）

**遇到问题时的处理流程：**
1. 完整记录错误现象（命令输出、错误信息、当前系统状态）
2. 尝试基于 AIMA 工具本身排查（不是绕过问题）
3. 如仍无法解决，停止该路径的测试
4. 汇总已获得的信息，向团队领导上报
5. 等待领导分析修复后再继续

## 二、AIMA 系统简介

AIMA（AI-Inference-Managed-by-AI）是运行在 AI 设备上的推理管理系统。它的核心能力：

- **硬件感知**：自动检测 GPU/CPU/内存等硬件配置
- **知识匹配**：根据硬件自动选择最优推理引擎和模型配置
- **一键部署**：单条命令部署推理服务，无需手动配置容器
- **多运行时**：支持 K3S（容器化）和 Native（本地进程）两种运行方式
- **统一 API**：对外暴露 OpenAI 兼容的推理接口

AIMA 采用渐进增强设计（L0→L3）：
- L0：内置默认配置，任何时候都能产出可用配置
- L2：知识库匹配，根据硬件自动选择最优配置
- L3：AI Agent 动态优化（此次测试不涉及）

**本次测试聚焦**：K3S 容器运行时 + VLLM 引擎 + Qwen 系列模型

## 三、AIMA 命令参考

### 系统状态与硬件检测

- `aima version`：确认 AIMA 版本，验证二进制可执行
- `aima status`：显示当前硬件信息（GPU/CPU/内存）+ 所有部署状态

### 基础设施初始化

- `aima init`：安装 K3S 和相关基础设施组件。仅支持 Linux，在 macOS/Windows 上会跳过
- `aima init --yes`：跳过下载确认，直接执行安装

### 引擎管理

- `aima engine scan`：扫描本地已有的推理引擎（容器镜像或本地二进制）
- `aima engine list`：列出 AIMA 数据库中已知的引擎
- `aima engine pull vllm`：拉取 VLLM 推理引擎的容器镜像
- `aima engine info <名称>`：查看某个引擎的详细信息

### 模型管理

- `aima model scan`：扫描本地文件系统，发现已有的模型文件
- `aima model list`：列出 AIMA 数据库中已知的模型
- `aima model import <路径>`：将本地路径下的模型文件导入 AIMA 管理
- `aima model pull <模型名>`：从网络下载指定模型
- `aima model info <模型名>`：查看模型的详细配置信息（包括知识库推荐配置）

### 知识库

- `aima knowledge list`：列出 catalog 中所有资产统计（硬件 profile、引擎、模型数量）
- `aima knowledge resolve <模型名>`：解析该模型在当前硬件上的最优配置
- `aima knowledge resolve <模型名> --engine vllm`：指定引擎解析配置

### 部署推理服务

- `aima deploy <模型名>`：部署推理服务（自动选择引擎）
- `aima deploy <模型名> --engine vllm`：指定使用 VLLM 引擎部署
- `aima undeploy <部署名>`：停止并删除某个部署

### 服务代理

- `aima serve`：启动 AIMA 推理代理服务（默认 HTTP 端口 6188）
- `aima serve --mcp`：同时启动 MCP 工具接口（端口 9090）
- `aima serve --discover`：自动发现局域网其他 AIMA 实例的模型并注册为远程 backend

## 四、本次测试涉及的目标模型

| 模型 | AIMA 模型名 | 规模 | 推理引擎 | 说明 |
|------|------------|------|---------|------|
| Qwen3-8B | `qwen3-8b` | 8B 全参数 | VLLM | 中等规模 LLM |
| Qwen3-30B-A3B | `qwen3-30b-a3b` | 30B MoE（3B 激活） | VLLM | MoE 架构，显存占用比参数量少 |
| Qwen3.5-35B-A3B | `qwen3.5-35b-a3b` | 35B MoE（3B 激活） | VLLM | GB10 专项性能测试 |
| Qwen3-ASR-1.7B | `qwen3-asr-1.7b` | 1.7B | CPU 推理 | 语音识别模型（场景 D 使用）|

**关于 MoE 模型**：Qwen3-30B-A3B 和 Qwen3.5-35B-A3B 虽然参数量大，但采用 MoE 架构，每次推理仅激活约 3B 参数，实际显存占用远小于同等规模的全参数模型。

## 五、测试场景

### 场景 A：K3S 基础设施初始化验证

**用户故事**：
作为一名运维工程师，我在一台全新的 Linux 服务器上首次部署 AIMA。我希望通过 AIMA 的初始化命令完成所有基础设施安装，安装完成后，系统能自动识别本机 GPU 资源并可用于推理任务。

**验收标准**：
- AIMA 二进制可正常运行，`version` 命令有输出
- `status` 命令能正确识别本机 GPU（厂商、型号、显存大小）
- `init` 命令执行完成后，K3S 相关组件显示为已就绪状态
- 知识库的硬件资产能匹配到当前机器的 GPU 型号

**测试步骤（按顺序执行）**：

1. **验证 AIMA 可执行**
   确认 AIMA 二进制文件已上传到 `~/aima` 路径，运行 `version` 命令，记录输出的版本号和构建时间。

2. **执行硬件检测**
   运行 `aima status` 命令，仔细阅读输出中的硬件信息部分。记录以下内容：GPU 厂商、型号、显存大小（MiB）、CUDA/ROCm 版本、CPU 核数、系统内存总量。

3. **验证知识库覆盖**
   运行 `aima knowledge list`，确认有硬件 profile、引擎资产、模型资产加载进来。尝试运行 `aima knowledge resolve qwen3-8b --engine vllm`，观察是否能找到匹配的配置。

4. **执行初始化**
   运行 `aima init`。如果有交互提示（询问是否下载），选择确认继续。等待安装完成，记录每个组件的安装结果（显示为 OK / FAIL / SKIP 状态）。

5. **验证初始化结果**
   初始化完成后，再次运行 `aima status`，查看 deployments 部分是否显示 K3S 已就绪。记录任何变化。

**预期障碍**：HAMi 组件可能在某些 GPU 上安装失败（这是已知问题，如遇到请如实记录错误信息但继续测试）。

### 场景 B：单模型 VLLM 部署验证

**用户故事**：
K3S 环境就绪后，我希望通过 AIMA 一条命令部署一个 Qwen3-8B 模型的 VLLM 推理服务，并验证 API 可以正常响应问答请求。

**前置条件**：场景 A 完成，K3S 至少部分就绪

**验收标准**：
- `aima deploy qwen3-8b --engine vllm` 能成功触发部署
- 部署完成后，`aima status` 显示该部署正在运行
- 启动 `aima serve` 后，向端口 6188 发送聊天请求，能收到模型的有效回复

**测试步骤**：

1. **检查模型状态**
   运行 `aima model list`，查看当前系统中已知的模型。如果 Qwen3-8B 已经在列表中，记录其路径。如果不在，需要先导入或下载。

2. **导入或扫描本地模型**
   如果机器上已有模型文件，使用 `aima model scan` 让 AIMA 发现它们，然后用 `aima model import <路径>` 导入。如果本地没有，使用 `aima model pull qwen3-8b` 从网络下载（记录下载耗时）。

3. **检查引擎准备情况**
   运行 `aima engine list` 查看已有引擎。运行 `aima engine scan` 扫描本地容器镜像。如果 VLLM 镜像不在列表中，运行 `aima engine pull vllm`（此步骤会拉取约 12GB 镜像，记录耗时）。

4. **解析配置（预演）**
   在实际部署前，运行 `aima knowledge resolve qwen3-8b --engine vllm`，查看 AIMA 计划使用的配置（GPU 内存利用率、最大序列长度等）。记录这些配置值。

5. **部署模型**
   运行 `aima deploy qwen3-8b --engine vllm`，等待部署完成。记录从命令发出到服务就绪的时间。部署完成后运行 `aima status` 查看部署状态。

6. **启动服务代理**
   运行 `aima serve` 启动 AIMA 的 HTTP 代理（在后台运行，注意保留其输出日志）。

7. **验证推理功能**
   使用你熟悉的 HTTP 工具，向 `http://localhost:6188/v1/chat/completions` 发送一个标准的 OpenAI Chat 格式请求（模型名使用 `qwen3-8b`，问一个简单问题如"你好，请自我介绍"）。记录完整的响应内容，包括推理耗时。

### 场景 C：双 VLLM 模型并行运行验证

**用户故事**：
在 Qwen3-8B 已在运行的情况下，我希望同时再部署一个 Qwen3-30B-A3B 模型，使两个推理服务并行运行，各自独立可用。

**前置条件**：场景 B 完成，Qwen3-8B 正在运行

**验收标准**：
- 两个模型的部署都显示为运行中
- 两个模型各自能独立响应推理请求
- 系统资源使用在可接受范围内（无 OOM）

**测试步骤**：

1. **确认当前状态**
   运行 `aima status`，确认 Qwen3-8B 服务正在运行。记录当前 GPU 显存使用量。

2. **准备 30B 模型**
   检查 `aima model list`，确认 Qwen3-30B-A3B 是否已知。如本地已有，确认路径正确。如没有，使用 `aima model pull qwen3-30b-a3b` 下载。

3. **解析 30B 模型配置**
   运行 `aima knowledge resolve qwen3-30b-a3b --engine vllm`，查看推荐配置。特别关注 `gpu_memory_utilization` 参数和所需最低显存。判断当前剩余显存是否够用。

4. **部署第二个模型**
   运行 `aima deploy qwen3-30b-a3b --engine vllm`，等待部署完成。注意：MoE 模型的首次加载可能需要较长时间。

5. **验证两个服务都可用**
   分别向 Qwen3-8B 和 Qwen3-30B-A3B 发送推理请求，确认两个服务都能正常响应。记录两个模型各自的响应速度（tokens/s）。

6. **记录资源状态**
   运行 `aima status` 查看两个部署同时运行时的资源使用情况。

### 场景 D：GPU LLM + CPU ASR 混合推理验证

**用户故事**：
在 GPU 上运行 Qwen3-30B-A3B 的同时，我希望通过 CPU 资源运行语音识别（ASR）模型，形成完整的多模态推理服务组合。

**注意**：此场景涉及 ASR 模型支持，AIMA catalog 中的 ASR 模型配置可能尚不完整。请如实记录遇到的任何问题，这是重要的发现。

**测试步骤**：

1. **查询 ASR 模型信息**
   运行 `aima model info qwen3-asr-1.7b`，查看 AIMA 对该模型的了解程度（是否有部署配置）。

2. **解析 ASR 部署配置**
   运行 `aima knowledge resolve qwen3-asr-1.7b`，查看 AIMA 是否能给出部署方案。记录完整输出。

3. **尝试部署 ASR 模型**
   尝试运行 `aima deploy qwen3-asr-1.7b`，观察 AIMA 的反应。如果失败，记录完整错误信息。

4. **记录当前能力边界**
   无论成功还是失败，记录以下信息：ASR 模型在 catalog 中的配置完整程度、AIMA 是否能处理非 LLM 模型的部署、错误信息（如有）。

### 场景 E：Qwen3.5-35B-A3B 性能基准测试（仅 GB10 机器）

**用户故事**：
在 GB10 机器上，对已下载好的 Qwen3.5-35B-A3B 模型进行全面性能测试，测量不同并发和上下文长度下的推理性能指标，为生产部署提供参考数据。

**目标指标**：
- TTFT（Time to First Token）：从请求到第一个 token 生成的延迟
- TPOT（Time Per Output Token）：每个 token 的平均生成时间
- 吞吐量（tokens/s）

**测试矩阵**：
| 测试维度 | 测试值 |
|---------|--------|
| 并发数 | 1、2、4、8 |
| 上下文长度（输入） | 512 tokens、2048 tokens、8192 tokens |

**测试步骤**：

1. **扫描并导入本地模型**
   使用 `aima model scan` 确认 AIMA 能发现机器上已有的 Qwen3.5-35B-A3B 模型文件。如未自动发现，使用 `aima model import <实际路径>` 手动导入。

2. **解析最优配置**
   运行 `aima knowledge resolve qwen3.5-35b-a3b --engine vllm`，记录推荐的配置参数，特别是 max_model_len 和 gpu_memory_utilization。

3. **部署模型**
   运行 `aima deploy qwen3.5-35b-a3b --engine vllm`，记录启动时间（从命令发出到推理就绪）。

4. **基准测试执行**
   启动 `aima serve`，然后对 API 端点进行性能测试：
   - 先进行单并发（1 个并发）下的基线测试：发送不同输入长度的请求，记录 TTFT 和 TPOT
   - 再逐步增加并发（2、4、8），观察性能变化趋势
   - 每个测试点至少发送 5 次请求，取平均值

5. **记录完整结果**
   将测试结果整理为表格形式，包含每个测试点的 TTFT 和 TPOT 数据。

### 场景 F：局域网服务发现验证

**用户故事**：
推理服务启动后，局域网内的其他设备和用户应该能够发现这个服务并使用它，无需手动配置 IP 地址。

**测试步骤**：

1. **确认服务已启动**
   运行 `aima serve --mcp` 启动 AIMA 的完整服务（包括 MCP 接口）。观察启动日志，查看是否有关于服务发现（mDNS）的输出。

2. **记录服务访问信息**
   记录以下信息供局域网用户使用：机器的 IP 地址、AIMA 服务端口（默认 6188）、已部署的模型名称。

3. **验证 API 可远程访问**
   如果条件允许，从局域网另一台设备（或使用同一机器的不同进程模拟）访问 `http://<机器IP>:6188/v1/chat/completions`，验证推理服务可用。

4. **探查 mDNS 服务发现状态**
   观察 `aima serve` 的日志输出，查看是否有 "_llm._tcp" 相关的 mDNS 广播信息。记录实际行为（已实现/未实现/报错）。

## 六、测试卡住的判断标准

遇到以下任何一种情况，停止该路径的测试，汇总并上报：

1. **命令报错且重试 2 次仍失败**：如 `aima deploy` 返回错误，记录错误信息，停止该场景
2. **等待超过 10 分钟无进展**：如镜像拉取长时间无输出，记录当前状态后停止
3. **系统状态异常**：如服务崩溃、OOM、机器无响应
4. **知识库无法匹配**：如 `aima knowledge resolve` 返回"未找到匹配配置"

**不应该视为卡住的情况**：
- 镜像拉取正在进行中（有进度输出）→ 等待完成
- HAMi 安装失败但 K3S 仍然可用 → 继续后续场景
- 模型启动需要 5-10 分钟 → 等待完成
