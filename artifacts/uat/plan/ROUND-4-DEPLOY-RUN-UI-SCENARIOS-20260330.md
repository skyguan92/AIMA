# Round-4 Deploy-Run-UI UAT Scenarios

**轮次**：Round-4  
**日期**：2026-03-30  
**构建**：`v0.2-dev` / commit `11f01e8`  
**方法**：按 `CLAUDE.md` 的 `ALL COLLECT, THEN ANALYZE` 原则执行，同一构建先收完整设备矩阵，再统一归并结果。  
**优先级**：功能第一，UI 第二。先验证真实部署链路是否成立，再看 UI 是否把真实状态表达清楚。

## 文档边界

- 本轮只看当前 `develop` 分支已经存在的 knowledge 和当前机器上已经存在的引擎/模型资产。
- 本轮不新增 catalog YAML，不修改代码，不为了“跑通”手工修环境。
- 场景文档只描述用户意图、观察点和通过信号；真实执行结果另记到 `sessions/`。
- 只要链路进入明确停止点，就记录下来；失败本身也是有效结果。

## UAT-DRU-01 这台机器现在会把我带到哪条部署路线

**用户故事**  
用户拿到当前 `develop` 构建后，不想先猜 runtime、引擎和模型格式。他希望 AIMA 基于当前 knowledge 和本机已有资产，先给出一条自洽的部署路线。

**重点观察**

- `hal detect`、`engine plan`、`model scan` 是否组成一条自洽的硬件与资产画像。
- `deploy --dry-run` 是否给出明确的 engine、runtime、model path/variant 方向。
- 同一台机器上，“已扫描到的资产”和“dry-run 计划使用的资产”是否一致。

**通过信号**

- 用户能在不猜底层实现的前提下，看见清晰的部署落点。
- 计划结果不自相矛盾，不出现“看得到资产、但计划完全不用它”的断裂。

## UAT-DRU-02 已有资产下的 deploy 能不能真的把实例推起来

**用户故事**  
用户已经确认当前机器有可用知识和资产，接下来他最在意的是 `aima deploy` 会不会真实触发部署，并把状态收敛到 ready 或明确 failed。

**重点观察**

- `deploy` 是否真的发出底层部署动作，而不是停留在解析层。
- `deploy status` 是否能追到同一个实例，而不是出现名字、runtime 或阶段漂移。
- 启动失败时，状态是否会诚实收敛到 `failed`，并带出足够的失败细节。

**通过信号**

- 部署要么进入 ready，要么进入明确 failed。
- 用户不会被 `starting`、`running but not ready` 这类长期中间态困住。

## UAT-DRU-03 已有实例或已有资产下的 run 语义是否成立

**用户故事**  
用户第二次执行 `aima run --no-pull` 时，期待的是“继续利用已有东西”，而不是重新回到下载链路，或者被一句底层冲突错误打回去。

**重点观察**

- `run --no-pull` 是否优先复用已有模型、已有引擎和已有 deployment。
- 若实例已经 ready，`run` 是否返回可用 endpoint，而不是只报 `already running`。
- 若实例未 ready 但资产已在，`run` 是否推进到明确的新状态。

**通过信号**

- `run --no-pull` 的语义更像“继续使用/推进当前服务”，而不是“另起一套不兼容流程”。
- 用户能拿到 endpoint、ready 状态，或者一条可判断的停止原因。

## UAT-DRU-04 `deploy` 与 `run` 的真实状态能不能被 UI 正确承接

**用户故事**  
用户打开 Web UI 时，不要求它比 CLI 更强，但至少应该忠实反映真实部署状态，让人一眼看懂当前是 ready、启动中，还是已失败。

**重点观察**

- `aima serve` 暴露的 `/ui/` 是否可进入，页面能否加载基础状态。
- Deployments 面板是否准确映射 CLI 看到的实例、phase、ready、endpoint 和失败细节。
- 当存在 `starting`、`failed`、`ready` 三种状态时，UI 是否表达清楚，而不是掩盖错误或给出误导性乐观状态。

**通过信号**

- UI 能把真实部署状态“如实翻译”给用户。
- 对同一实例，CLI 与 UI 在名称、阶段、可访问地址上的表达一致。

## UAT-DRU-05 UI 整体体验是否至少不妨碍核心任务

**用户故事**  
在核心功能成立的前提下，用户希望 UI 至少具备可进入、可辨识、可操作的基础质量，而不是在视觉或交互上妨碍判断。

**重点观察**

- 页面首屏信息层级是否清楚，是否能快速找到部署状态和入口。
- 桌面端与移动视口下，主要信息是否仍可读，不出现明显遮挡或错位。
- 功能异常时，错误文案和状态色是否帮助判断，而不是制造更多歧义。

**通过信号**

- UI 虽然不是本轮第一优先级，但不会拖累功能验收。
- 用户能依靠 UI 做基本判断，不必被迫回到控制台才能理解当前状态。
