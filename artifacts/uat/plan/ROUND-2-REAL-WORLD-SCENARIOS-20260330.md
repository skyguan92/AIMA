**轮次**：Round-2  
**日期**：2026-03-30  
**目标**：围绕最近的大改动，验证真实用户最关心的“下载指定版本引擎”和“一条命令跑起模型”两条链路。

## UAT-RW-01 指定版本引擎下载

**用户故事**  
用户已经知道自己想要的引擎版本，例如某个 CUDA / vLLM 版本。他只想告诉 AIMA “我要这个版本的引擎”，然后等待下载完成，不想自己判断该拉哪个镜像、该进 Docker 还是 containerd、是否还要补一轮导入。

**重点观察**

- AIMA 是否能为当前硬件解析到正确的引擎资产和版本。
- 下载完成后，引擎是否真的进入“可用”状态，而不是只存在于某个用户并不理解的底层存储里。
- 失败时，AIMA 是否明确告诉用户卡在哪个 registry / runtime / 权限边界。

**通过信号**

- 用户确认版本号正确。
- 用户无需补做底层镜像导入动作。
- 用户能直接把这个引擎用于后续 `run` / `deploy`。

## UAT-RW-02 一条命令从空目录跑起模型

**用户故事**  
用户在一个干净的数据目录里第一次运行某个模型，希望 `aima run <model>` 自动完成硬件识别、引擎解析、缺失引擎下载、缺失模型下载、部署和就绪等待，最后给出可访问的服务地址。

**重点观察**

- AIMA 是否能选到符合当前机器的默认引擎和 runtime。
- 缺失资产是否会被自动补齐，而不是在中途要求用户切换到别的命令。
- 链路失败时，失败点是否清晰，是否会在“半完成状态”里把用户困住。

**通过信号**

- 用户只发起一次 `run` 意图。
- 链路能进入 ready 状态并返回 endpoint。
- 用户无需自己处理底层镜像、模型目录或额外注册动作。

## UAT-RW-03 部分资产已存在时的重试体验

**用户故事**  
用户已经有一部分资产，例如镜像已经在 Docker 里、模型下载到一半、或者上一次 `run` 被中断。他再次发起同样的运行意图时，希望 AIMA 能延续之前的状态，而不是要求他理解内部差异并手工补救。

**重点观察**

- AIMA 是否能识别“已经有一部分东西”。
- 是否会自动选择继续、复用、导入、恢复，而不是重新从零开始。
- 用户是否必须了解 Docker / containerd / Hugging Face / ModelScope 的内部差别。

**通过信号**

- 第二次尝试比第一次更接近完成，而不是回到起点。
- 中断后的状态不会把用户锁死在某个内部实现细节上。

## UAT-RW-04 大模型下载不稳定网络恢复

**用户故事**  
用户在真实公网环境下载大模型时，网络可能抖动、超时、切源或中断。他希望 AIMA 把这些不稳定因素吸收掉，而不是把一条本该自动化的链路变成漫长的重试和人工判断。

**重点观察**

- 下载失败后是否会自动恢复，而不是整条链路直接报错退出。
- 切源后是否能复用已完成的数据，而不是重新做一轮大下载。
- 用户能否在合理时间里知道这次运行还有没有希望完成。

**通过信号**

- 下载异常后可以继续推进。
- 已完成的数据被保留并复用。
- 用户能收到清晰、可判断的状态反馈。
